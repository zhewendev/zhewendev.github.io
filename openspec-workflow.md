# OpenSpec Workflow

> 本文档定义项目中 OpenSpec 的完整工作流程。所有 AI Agent 使用 `/opsx:explore`、`/opsx:propose`、`/opsx:apply`、`/opsx:archive` 时必须遵循此流程。

## 核心原则

1. **持久化优先** — 讨论中达成的任何确定结论必须立即写入文件，不依赖对话记忆。
2. **分阶段执行** — 实施按 Phase 推进，每阶段验证通过后方可进入下一阶段。
3. **最小修改** — 每次只做当前阶段需要的事，不跨阶段操作。
4. **Blueprint 合规** — 所有变更须符合 Blueprint 规则，冲突时暂停并报告。

---

## 阶段一：Explore（需求探索）

### 目标

理解需求、澄清边界、形成经用户逐项确认的需求清单。

### 增量持久化机制

Explore 过程中（尤其是逐页讨论设计稿时），上下文持续增长，后期生成的 Proposal 容易因上下文溢出丢失早期讨论细节。解决方式：**每确认一项需求，立即持久化并请用户确认。**

#### 操作步骤

1. **创建变更目录** — 当探索方向明确后，立即创建：
   ```bash
   openspec new change "<name>"
   ```

2. **创建探索笔记** — 在变更目录下创建 `explore-notes.md`，作为讨论结论的持久化载体，是后续 Proposal 生成的唯一输入源。

3. **逐项讨论 → 确认 → 持久化** — 每完成一个需求/页面的讨论并达成结论时：
   - 将结论结构化写入 `explore-notes.md`
   - 向用户确认：**"已记录 [需求名称] 的讨论结论，是否保持？"**
   - 用户确认 → 继续下一项；用户要求修改 → 当场更新笔记后再次确认

4. **全部讨论完毕** — 向用户展示 `explore-notes.md` 完整摘要，确认无误后进入 Propose。

#### 探索笔记格式

```markdown
# Explore Notes: <change-name>

## 需求清单

### 1. <需求/页面名称>
- **描述**: 一句话说明
- **关键决策**: 讨论中确定的实现选择
- **涉及组件/文件**: 需要新增或修改的部分
- **Blueprint 参考**: 引用的 Blueprint 章节（如适用）
- **开放问题**: 待解决项（如有）

### 2. <需求/页面名称>
...

## 全局决策
- 跨页面的共同约定（设计风格、交互模式等）

## 开放问题
- 待解决的全局问题
```

### 上下文预算意识

- 对话变长时，主动提示："当前讨论较长，建议持久化已确定结论后继续。"
- 优先讨论核心页面，次要页面可简化。
- 如发现已有结论可能在后续讨论中被遗忘，立即写入笔记。

### 探索结束条件

- 所有需求/页面已讨论完毕
- `explore-notes.md` 中每项均已用户确认
- 无阻塞级开放问题（非阻塞问题可在 Proposal 中标记为 TBD）

## 阶段二：Propose（方案生成）

### 目标

基于 `explore-notes.md` 生成正式的 OpenSpec 文档：`proposal.md`、`design.md`、`specs/`、`tasks.md`。

### 关键约束

**文档生成必须基于 `explore-notes.md`，不依赖对话记忆。**

- 先完整读取 `explore-notes.md`。
- 如发现笔记中有不明确之处，向用户确认，不自行猜测。
- 生成过程中如发现遗漏，暂停并补充探索。

### 生成流程

按 OpenSpec 标准流程执行（`openspec instructions <artifact>` 逐个生成），增加以下检查点：

1. **Proposal 生成后** — 与 `explore-notes.md` 逐项对照，确认所有需求均已覆盖。有遗漏则补充。
2. **Tasks 生成后** — 确认任务已按 Phase 分组（见下方规则）。

### Phase 定义规则

`tasks.md` 中的任务必须按 Phase 分组，每个 Phase 是一个可独立验证的交付单元：

```markdown
## Phase 1: <phase 名称>
- [ ] task 1
- [ ] task 2

## Phase 2: <phase 名称>
- [ ] task 3
- [ ] task 4
```

划分原则：
- 每个 Phase 产出可验证的结果（页面可预览、功能可测试、类型检查通过等）。
- Phase 间存在自然依赖（如先布局后交互、先结构后样式）。
- 数量适中，通常 2-5 个，避免过细或过粗。

### Blueprint 合规检查

所有文档生成后、进入 Apply 前，执行一次 Blueprint 合规检查：
- 变更内容是否与 Blueprint 冲突。
- 如有冲突：**立即暂停，报告冲突并等待用户指示。** 不得自行修改 Blueprint。

### 文档审查

向用户展示文档摘要并请求确认。用户确认后方可进入 Apply。

---

## 阶段三：Apply（分阶段实施）

### 目标

按 Phase 逐步实现 `tasks.md` 中的任务。

### 执行规则

1. **默认逐 Phase 执行** — 一次只执行一个 Phase 的所有任务。
2. **Phase 完成后必须验证** — 执行验证检查（见下方清单）。
3. **验证通过后询问用户** — "Phase N 已完成并通过验证，是否继续执行 Phase N+1？"
4. **用户未确认前不继续** — 除非用户明确要求一次执行所有 Phase。

### 验证清单

每个 Phase 完成后，按项目实际适用项执行检查：

| 检查项 | 命令 | 适用场景 |
|--------|------|----------|
| 类型检查 | `pnpm astro check` | 涉及 TS/TSX 文件 |
| 代码规范 | `pnpm run lint` | 涉及代码修改 |
| 格式检查 | `pnpm run format:check` | 涉及代码修改 |
| 格式修复 | `pnpm run format` | 格式检查未通过时 |
| 完整构建 | `pnpm run build` | 涉及构建相关变更 |
| 视觉验证 | 用户在浏览器中确认 | 涉及 UI 变更 |

验证流程：
1. 运行适用的检查命令。
2. 如有错误，当场修复（同一文件最多 3 次尝试）。
3. 修复失败则暂停，报告问题并等待用户指示。
4. 全部通过后，向用户汇报验证结果。
5. 询问是否进入下一 Phase。

### 一次性执行所有 Phase 的处理

如果用户明确要求一次执行所有 Phase：

1. 仍按 Phase 顺序执行，不跳过 Phase 边界。
2. 每个 Phase 完成后运行验证。
3. 如某 Phase 验证失败，**暂停并报告，不继续后续 Phase**。
4. 全部完成后，执行一次完整的端到端验证（类型检查 + lint + 构建）。
5. 向用户汇报完整的验证结果。

### 暂停与恢复

实施可在任意 Phase 边界暂停。恢复时：
- 重新读取 `tasks.md` 确认进度。
- 从下一个未完成的 Phase 继续。

### 变更回退

如某 Phase 验证失败且无法快速修复：
- 暂停实施。
- 报告失败原因和受影响文件。
- 提供选项：回退该 Phase 修改 / 修复后继续 / 更新方案后继续。
- 由用户决定下一步。

---

## 阶段四：Archive（归档）

按标准 OpenSpec Archive 流程执行。归档前确认：
- 所有任务已完成（`- [x]`）。
- 最终验证全部通过。
- Delta specs 已同步（如适用）。

`explore-notes.md` 随变更目录一起归档，作为需求追溯依据。

---

## 流程速查

```
Explore                         Propose                        Apply                         Archive
  │                               │                              │                              │
  ├─ 创建变更目录                  ├─ 读取 explore-notes          ├─ 逐 Phase 执行               ├─ 确认任务完成
  ├─ 创建 explore-notes           ├─ 生成 proposal               ├─ 每 Phase 验证               ├─ 同步 delta specs
  ├─ 逐项讨论 → 写入 → 确认保持    ├─ 生成 design                 ├─ 询问是否继续                ├─ 归档
  ├─ 上下文过长时主动提示持久化     ├─ 生成 specs                  │                              │
  ├─ 全部完成后展示摘要确认         ├─ 生成 tasks（按 Phase 分组）  │                              │
  │                               ├─ Blueprint 合规检查           │                              │
  │                               └─ 用户确认                     │                              │
  ▼                               ▼                              ▼                              ▼
```

## 关键约束汇总

| 约束 | 说明 |
|------|------|
| Explore 增量持久化 | 每确认一项需求，立即写入 `explore-notes.md` 并请用户确认保持 |
| Proposal 基于笔记 | 文档生成以 `explore-notes.md` 为唯一输入源，不依赖对话记忆 |
| Tasks 按 Phase 分组 | 每个 Phase 是可独立验证的交付单元 |
| 逐 Phase 执行 | 默认一次只执行一个 Phase，验证通过后询问用户是否继续 |
| Phase 验证必须 | 每个 Phase 完成后必须运行验证检查，失败则暂停 |
| 一次性执行仍验证 | 即使用户要求一次执行所有 Phase，每个 Phase 仍需验证，全部完成后做端到端验证 |
| Blueprint 合规 | Proposal 生成后检查 Blueprint 合规性，冲突时暂停报告 |
