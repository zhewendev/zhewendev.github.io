> 定义内容从创建、编辑、发布到长期维护的全生命周期。本文档描述工作流，不涉及技术实现。

---

## 一、核心原则

- **事实来源**：Markdown 为唯一事实来源（Single Source of Truth）。
- **工具链**：本地优先、Markdown 优先、Git 优先。
- **协作**：AI 辅助创作，但不替代作者思考。
- **迭代**：内容发布后持续演进，而非一次性完成。

## 二、内容生命周期

所有内容均遵循统一生命周期。
统一路径：灵感 → 创建 → 完善 → 发布 → 维护 →（归档）

| 阶段  | 说明                    |
| --- | --------------------- |
| 灵感  | 收集想法、链接、笔记            |
| 创建  | 新建文件，填写基础 Frontmatter |
| 完善  | 完成正文及全部元数据            |
| 发布  | 合并至主分支，CI 自动部署        |
| 维护  | 持续更新内容与关联             |
| 归档  | 内容失效时保留历史             |

## 三、内容来源

1. **Obsidian**：主要写作入口（文章、随想、笔记、工具、专题）。
2. **AI 协作**：大纲生成、润色、错误检查、元数据补全、标签建议、SEO 优化。
3. **外部资源**：阅读摘录、文档、开源项目等，须注明来源，禁止直接复制。

## 四、创建流程

1. **确定类型**（Article / Series / Book / Tool / Thought / Friend / About），发布后不修改。
2. **创建文件**，填写基础 Frontmatter（标题、Slug、发布时间、分类、标签），正文可为空。
3. **完成正文**（标题层级、图片、引用、代码块、表格），优先保证可读性。
4. **完善元数据**（description、tags、cover、modDatetime 等）。
5. **提交发布**：Git 提交 → CI 自动构建、检查、生成 Sitemap/RSS、部署（无需人工操作）。

## 五、修改流程

修改 Markdown → 更新 `modDatetime`（必要时更新 description/tags）→ 提交 Git → 自动部署。
确保正文与 Frontmatter 保持一致。

## 六、特定类型维护

- **专题（Series）**：创建专题并填写介绍，关联文章。新增文章指定所属专题后自动出现。调整顺序仅修改引用关系，不修改文章内容。
- **收藏（Book / Tool）**：创建资源并填写信息，可选关联文章。支持长期更新（如评分、官网、关联文章）。
- **随想（Thought）**：保持轻量，快速记录与发布。若内容逐步形成体系，迁移为 Article，迁移后保留原发布时间，更新修改时间。

## 七、内容质量要求

- **完整**：包含完整上下文，不只是结论。
- **可维护**：禁止"以后再补"占位；长期未完成的保持 Draft。
- **可检索**：标题明确、分类准确、标签合理、description 清晰。
- **可关联**：尽可能建立专题、相关文章、收藏等链接，形成内容网络。

## 八、Git 工作流

- 所有变更通过 Git 管理。
- 流程：本地修改 → Commit → Push → GitHub → GitHub Actions → 构建 → 部署。
- Git 历史即内容历史。
- **禁止**直接在线修改已发布内容。

## 九、AI 协作原则

**AI 可执行**：
- 生成大纲、补充元数据、检查格式、优化表达、推荐标签与相关文章。

**AI 禁止**：
- 捏造事实、修改作者观点、删除重要内容、编造引用来源。

所有最终内容须由作者确认。

## 十、长期维护原则

1. 优先更新旧文章，而不是重复写新文章。
2. 保持内容长期有效。
3. 定期检查失效链接。
4. 定期补充相关推荐。
5. 保持 Frontmatter 与正文一致。
6. 所有内容均可持续迭代，而非一次性完成。

## 十一、总则

> 通用原则见 `00-overview` §原则与理念，AI 协作边界见本文档 §九。以下为内容工作流的工具链总则。

1. Markdown 是唯一事实来源。
2. Obsidian 是唯一写作入口。
3. Git 是唯一版本管理工具。
4. GitHub 是唯一远程仓库。
5. GitHub Actions 是唯一发布入口。
